Articles I've Read

Will try to keep this updated, and maybe even jot down an inane thought or two.

  • What Will Save Us from the Dark Side of CSS Pre-Processors? by Lyza Danger Gardner, but also A Vision for Our Sass by Felicity Evans and even The problem with CSS pre-processors by Miller H. Borges Medeiros (from 2011!). Though realistically I also read a bunch of stuff by Harry Roberts. ... among other articles on SCSS.

    Read on Wed Mar 9 2016.

    As a new user of SCSS I hadn't thought enough about how mixins and extends can negatively affect filesize by creating duplicate styles. What seems dry can be just the opposite when the mixins I wrote once in SCSS then appear over and over again in my CSS... perhaps not ideal? Harry Roberts does point out here how what we should be after is a "single source of truth" rather than avoiding repetition in the compiled css. I haven't formed a complete opinion here myself. This is covered here.

    Tight coupling is also worth being aware of - any styles that are used as extensions have to be documented as such, lest edits in one style lead to unintended changes in the styles that the edited one feeds into.

    This article by Harry Roberts goes even further in pointing out the perils of @extend, namely, that it extends all instances of the class, which is perhaps undesirable, and maybe worth avoiding entirely. The proposed solution involves silent classes, written like so

    .foo,
%foo {
  color: #acce55;
}

.bar .foo {
  background-color: #0dba11;
}

.baz {
  @extend %foo;
}

    As this will allow us to avoid extending the background-color into baz, instead extending only the color property.

    And, of course, Harry continues in this line of unpacking the perils of `@extend` in a much more recent article. Where he points out that after gzipping, the repetition from mixins actually resolves to a smaller, more efficient filesize than even @extends that result in smaller .css files sizes due to comma separated classes. Somewhere along the way I also picked up the possibility of setting default values for mixins like this: @mixin my-mixin($var: default) {} which could be handy!

    The Sass Style Guide by Chris Coyier is also a useful resource. Something that I already got out of it is the advantage of using mixins for media queries. This makes nesting media queries in the classes that get affected by them much neater and it becomes an actually desirable strategy. This works because we keep account of the breakpoints in a centralized place and if we want to move these around we can do so without having to hunt down each media query nested in the classes. It also mentions the use of the funny but quite likely very useful shame.css file to help with, eventually, cleaning up hacky code. The first step for me will of course be recognizing what is hacky and what is solid. `!important` and IDs some to mind... but apart from that?

    Ethan Muller has an article called Naming CSS Stuff Is Really Hard where he recommends using functional class names (like "positive-button") that are less likely to change instead of getting too specific. This isn't always possible due to incidental styles. He argues against using content based classes (like "submit-button") for larger sites and and espouses presentational styles like ("green-button") as these communicate more clearly and allow for more reusability.

    Felicity Evans in A Vision for Our Sass mentions the need to name variables based on their function rather than the way they look. This makes sense when thinking about changing the style of a page so that you can avoid having to rename all the classes each time you change a theme color. There is no easy answer between this and the points Ethan Muller made, but navigating through this seems like a reasonable challenge to undertake in my css experience. Of course she goes on an makes a convincing case against presentational class names as well, urging to use these as @extends instead, and as such keeping them out of the HTML.

    The first takeaway from Lyza Danger Gardner in What Will Save Us from the Dark Side of CSS Pre-Processors? is to actually look at the outputed CSS when using SCSS... this has been mentioned by many of the articles linked before as well. The reason I chose this article in the first place (it was really my starting point but it ended up branching to all the other stuff I read today/yesterday) was to get a sense of PostCSS... missed the mark on that, but I learned a lot about preprocessors! PostCSS will come soon.

  • All Talk and No Buttons: The Conversational UI by Matty Mariansky

    Read on Sun Mar 6 2016.

    This and the follow up article talk about an alternative to the UX we're used to. Basically an AI driven experience where a user types (or says, though Matty seems to be less into that) commands and the program parses them intelligently. This requires some rethinking in the kind of discoverability alloted to users. Pretty interesting. While the case for why this is better wasn't expressly communicated, I still think the possibilities and accessibility of such a UI is great. Recommended!
